最佳实践：从氛围式编码到代理式工程

第一支视频：命令、代理与技能如何串成工作流

引子

问题在哪里？

如果你只是输入提示词、看结果、再来一轮，那么你只用到了 Claude Code 的一小部分能力。

氛围式编码 vs 代理式工程

氛围式编码

打一段提示，拿到一个结果，然后继续追问。

它能工作，但过程不稳定，也不容易复现。

你始终被困在回合里，必须盯着它一步步做完。

代理式工程

把流程定义一次，之后反复复用。

命令、代理和技能彼此协同，形成可重复执行的路径。

你发起任务后，可以让它按既定方式自动跑完。

先看反例：随手问天气

自由提示词

在一个新的 Claude Code 终端中输入：

> 迪拜现在多少度？把结果写进输出文件，再生成一张 SVG 天气卡片。

问题

它也许能给你结果，但颜色、布局、输出路径和文件命名很可能每次都不一样。

第 1 部分

工作流方式

把同一个任务写成一条可重复、可自动执行的命令。

/weather-orchestrator

触发命令

与其自由描述，不如直接运行：

> /weather-orchestrator
1
先询问摄氏还是华氏用户输入被结构化收集，而不是让模型临场猜。
2
派生天气代理由专门的代理去获取温度和数据。
3
调用 SVG 技能固定生成统一风格的天气卡片。
4
稳定输出始终生成同一路径下的 weather.svg 和 output.md。

第 2 部分

它到底是怎么串起来的？

核心链路就是：命令 → 代理 → 技能。

三个构件

命令
命令位于 .claude/commands/，负责收集输入并编排后续步骤。
代理
代理位于 .claude/agents/，处理更专门的任务，例如查询天气。
技能
技能位于 .claude/skills/，既可以预加载给代理，也可以在运行时单独调用。

关键理解

命令负责“编排”，代理负责“执行”，技能负责“沉淀可复用知识与动作”。

快速参考

概念位置作用
命令.claude/commands/入口、编排、用户交互
代理.claude/agents/专门工种，带有自己的工具和职责
技能.claude/skills/可复用的知识包或执行说明
