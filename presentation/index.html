Claude Code

从氛围式编码到代理式工程

用一个真实仓库，串起从入门到高阶的最佳实践。

示例项目：TodoApp

贯穿示例

整套演示都围绕一个 TodoApp 单体仓库 展开，展示项目如何从“随手问一句”逐步变成“具备稳定工作流的工程项目”。

之前：氛围式编码

# 只是一个普通项目 todoapp/ backend/ frontend/ README.md

之后：代理式工程

# 接入 Claude Code 最佳实践 todoapp/ .claude/ agents/ skills/ commands/ hooks/ settings.json CLAUDE.md

什么是氛围式编码？

低水平使用

> 给我加一个笔记功能

它会写出能跑的代码，但结构、风格、测试都不稳定。

新页面可能不接入导航，新接口可能不沿用已有模式。

每一次提示都像重新掷一次骰子。

我们真正想要的

Claude 知道项目结构、边界、约定和测试习惯。

新功能能沿用已有模块、命名和工作流。

输出稳定、可复用、可验证。

这就是代理式工程的起点。

这趟旅程会经过什么？

1更好的提问方式
2项目记忆与规则
3结构化工作流
4领域知识与技能
5多代理协作
6收尾与验证

第 1 阶段

更好的提问方式

明确目标、文件范围、验收标准和约束条件，能立刻把成功率从“碰运气”提升到“有方向”。

第 2 阶段

项目记忆与规则

通过 CLAUDE.md、路径规则和项目上下文，让 Claude 不只是回答问题，而是理解代码库。

第 3 阶段

结构化工作流

把经常重复的操作沉淀成命令、步骤和可复用流程，让结果稳定可复现。

第 4 阶段

领域知识与技能

把 API 习惯、业务术语、图形输出规则和安全边界封装成技能，减少每次重复解释。

第 5 阶段

多代理协作

把查资料、写实现、做验证拆给不同角色，形成可靠的并行工作方式。

你可以立刻开始的 5 件事

1
先跑一次 /init生成最基础的项目上下文，让 Claude 先认识你的仓库。
2
对真实任务使用计划模式先看步骤，再让它落地，能显著降低返工。
3
把重复流程写成技能例如生成 SVG、跑测试、做发布检查。
4
为关键目录写局部说明前端、后端、基础设施可以有各自的 CLAUDE.md。
5
完成后做验证把“能跑”升级成“有证据地完成”。

黄金规则

复杂任务先计划，说明文件保持精炼，尽量一次只推进一个明确目标，并在完成前运行验证。
